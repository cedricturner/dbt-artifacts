below is an example of a stored procedure. it builds a customer order summary. highlighted lines are the things that create problems at scale. read the inline labels, then move to the next tab to see how dbt solves each one.
build_customer_order_summary, full stored procedure 1 file · ~60 lines 
-- ================================================================
-- Stored Procedure: build_customer_order_summary
-- Runs nightly. Touch with care.
-- ================================================================

CREATE OR REPLACE PROCEDURE build_customer_order_summary()
AS $$
BEGIN

  -- Step 1: clean customers
  CREATE TEMP TABLE tmp_customers AS  ⚠ temp, gone when session ends, nothing to inspect on failure
  SELECT
      customer_id,
      TRIM(LOWER(email))               AS email,
      COALESCE(first_name, 'Unknown')  AS first_name,
      COALESCE(last_name,  'Unknown')  AS last_name,
      CASE WHEN region IS NULL THEN 'Unassigned' ELSE UPPER(region) END AS region,
      created_at
  FROM raw.customers;  ⚠ hardcoded schema, rename silently breaks this

  -- Step 2: clean orders
  CREATE TEMP TABLE tmp_orders AS  ⚠ temp, same problem
  SELECT
      order_id, customer_id, order_date, status, total_amount
  FROM raw.orders
  WHERE order_id IS NOT NULL AND status != 'cancelled';

  -- Step 3: clean payments
  CREATE TEMP TABLE tmp_payments AS  ⚠ temp, same problem
  SELECT
      payment_id, order_id, payment_method,
      amount AS payment_amount, created_at AS payment_date
  FROM raw.payments
  WHERE payment_id IS NOT NULL AND amount > 0;

  -- Step 4: join orders + payments
  CREATE TEMP TABLE tmp_orders_with_payments AS  ⚡ must wait for steps 2 & 3, no parallelism possible
  SELECT
      o.order_id, o.customer_id, o.order_date, o.status, o.total_amount,
      SUM(p.payment_amount)                            AS total_paid,
      COUNT(p.payment_id)                               AS payment_count,
      o.total_amount - COALESCE(SUM(p.payment_amount),0) AS outstanding_balance
  FROM tmp_orders o
  LEFT JOIN tmp_payments p ON o.order_id = p.order_id
  GROUP BY o.order_id, o.customer_id, o.order_date, o.status, o.total_amount;

  -- Step 5: final output
  CREATE OR REPLACE TABLE analytics.fct_customer_order_summary AS  ⚠ full overwrite every run, no incremental option
  SELECT
      c.customer_id, c.email,
      c.first_name || ' ' || c.last_name AS full_name, c.region,
      COUNT(op.order_id)                   AS lifetime_orders,
      COALESCE(SUM(op.total_amount),0)   AS lifetime_revenue,
      COALESCE(SUM(op.total_paid),0)     AS lifetime_payments,
      MAX(op.order_date)                   AS most_recent_order_date
  FROM tmp_customers c
  LEFT JOIN tmp_orders_with_payments op ON c.customer_id = op.customer_id
  GROUP BY c.customer_id, c.email, full_name, c.region;

  -- Proc ends. All 4 temp tables dropped automatically.
  -- If step 5 failed: no intermediate state survives to inspect.  ⚠ debug from scratch

END;
$$ LANGUAGE plpgsql;
this stored procedure has five steps in one file. each step runs in order, schema names are hardcoded, and every run overwrites the full table. if it fails, the temporary tables are gone, so there is no intermediate state to inspect. the next tab shows how dbt handles the same work.

You've now seen all 5 steps. The SQL didn't change meaningfully. What changed is the structure: each step is isolated, independently runnable, and inspectable. The next tab explains what the naming conventions mean and how dbt tracks where data comes from.
the SQL stays the same. dbt changes how each step is represented: CREATE TEMP TABLE becomes a model file, raw.x becomes source(), and tmp_x becomes ref(). these changes let dbt track dependencies, run models separately, and preserve intermediate results.
📂
the prefixes show where each model sits in the pipeline. they also make the lineage graph easier to read by showing where data enters and how it moves through dbt.
stg_
Staging, one file per source table. Cleans, renames, casts. Raw in, usable columns out. Nothing is joined here.
int_
Intermediate, joins and transforms that aren't the final output. Shared building blocks between mart models.
fct_ / dim_
Marts, what BI tools and business users actually query. One table per business question.

source(), where raw data enters

raw.customers→{{ source('raw', 'customers') }}
raw.orders→{{ source('raw', 'orders') }}
raw.payments→{{ source('raw', 'payments') }}
The stored proc hardcoded raw.customers. If that schema renames, the proc breaks silently. source() registers these as official entry points, dbt can test them, check freshness, and alert you before a bad data day becomes a bad dashboard day.

ref(), how models depend on each other

FROM tmp_orders→FROM {{ ref('stg_orders') }}
FROM tmp_payments→FROM {{ ref('stg_payments') }}
FROM tmp_customers, tmp_orders_with_payments→ref('stg_customers'), ref('int_orders_with_payments')
Every ref() is a declared dependency. dbt reads them all and builds the execution order automatically. You don't write orchestration code, the ref() calls are the orchestration.

the graph shows the structure that was buried in the stored procedure. step 4 depends on steps 2 and 3, but you had to find that relationship by reading 60 lines of SQL. dbt records the dependency through ref(), which makes it visible in the graph and available in generated documentation.
📓
One setup step makes source(), lineage, and testing work. You tell dbt where your raw data lives, once, in a small YAML file, and you attach tests right there. After this, the tables you saw the proc read from become declared, tested entry points.
models/staging/_sources.yml define raw data once 
# Declare your raw data, and test it.
version: 2

sources:
  - name: raw                 # now usable as source('raw', ...)
    schema: raw
    tables:
      - name: orders
        description: Raw orders, loaded by your EL tool.
        freshness:               # warn if the feed goes stale
          warn_after: {count: 24, period: hour}
        columns:
          - name: order_id
            tests:
              - unique
              - not_null
          - name: status
            tests:
              - accepted_values:
                  values: [placed, shipped, cancelled]
      - name: customers
        columns:
          - name: customer_id
            tests: [unique, not_null]
      - name: payments

What this one file gives you

  • Reference by name: write source('raw','orders') instead of hardcoding raw.orders.
  • Tests that run every build: unique, not_null, accepted_values, failing fast before bad data spreads.
  • Docs and freshness: the description and freshness rule feed documentation and lineage automatically.

Same idea, on your models

  • Models get their own schema.yml with the same kind of tests.
  • So tests sit on sources and models, right next to the SQL.
  • It is all code, so it is reviewed and version-controlled in git like everything else.

How does dbt know it can parallelize? Because stg_customers, stg_orders, and stg_payments each only reference source(), they have no ref() edges between them. dbt reads the graph, sees no dependencies, and dispatches all three at once. The ref() declarations from tab 2 are the orchestration, you didn't write any extra config to make this happen.
At scale: This example has 5 models. Real projects have hundreds. A pipeline with 40 independent staging models runs in roughly the time it takes to run one, given enough threads.
🔬
The stored proc works until it doesn't. When it breaks, dbt leaves you with more to inspect and less to rerun. Click each scenario to compare.
Pipeline fails at the join step (step 4)▼
stored procedure

Step 4 failed. The temp tables from steps 1–3 died with the session. To reproduce the failure, run the entire procedure again from the top.

dbt

Step 4 failed. stg_orders and stg_payments are still in the warehouse. Run dbt run -s int_orders_with_payments to re-execute only that model. Upstream work stays intact.

dbt models remain in your warehouse after a failed run. Stored proc temp tables disappear with the session.
Wrong row count in the final output▼
stored procedure

Fan-out in the join? Bad filter in step 2? You add temp debug tables mid-proc, re-run everything, and manually query each one. Multiple cycles to isolate.

dbt

Count the rows in stg_orders, then count them in int_orders_with_payments. The difference shows where the fan-out happened. You don't need to rerun either model.

Because every intermediate model is a real table, you can bisect a row count problem the same way you'd bisect a bug in code, without re-running anything.
Upstream schema change, a column got renamed▼
stored procedure

The proc either silently returns NULLs or throws an error. You find out when a BI dashboard breaks. Trace back through the proc to find which temp table was affected.

dbt

stg_payments is the only model that touches raw.payments. Fix it there. Every downstream model gets the fix automatically on the next run. One change point, not five.

The stg_ layer acts as an isolation boundary. Raw schema changes only affect one file, the staging model for that source.
3am on-call alert fires▼
stored procedure

Connect to the DB. No intermediate state anywhere. Re-run the full proc to reproduce. Add RAISE NOTICE statements. Re-run again. 45 minutes of archaeology before you even start fixing.

dbt

dbt run logs show the exact model that failed and the exact error. Intermediate models are still in the warehouse. Run only the failed model, fix, run downstream only.

At 3am, saved intermediate tables give the on-call engineer somewhere useful to start.
What else depends on orders data?▼
stored procedure

Grep the codebase and hope nothing uses tmp_orders in another proc you don't know about. No system-of-record for dependencies.

dbt

The lineage graph lists every downstream model that references stg_orders. Click the model in the dbt DAG to see what a change could affect.

A new engineer can answer "what depends on this?" in 5 seconds with the DAG instead of 30 minutes of codebase archaeology.
The practical difference: dbt leaves intermediate models in the warehouse as queryable tables. Stored proc temp tables disappear when the session ends.
