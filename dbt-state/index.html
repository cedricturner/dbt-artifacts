dbt State, explained

You would not buy a ticket to watch the same movie again, the one you have already seen, where nothing has changed. (Sure, you might re-watch one you are emotionally attached to. But nobody rebuilds a data model out of sentimentality.) You go back to the theater when there is a new release.

By default, dbt does not work that way. Every time it runs, it rebuilds every model, whether or not anything upstream actually changed. At scale, with jobs running many times a day to keep dashboards fresh, that is a lot of tickets bought for movies you have already seen.

dbt State fixes that. Its whole idea fits in six words: build what's changed, skip what hasn't.

THE WASTE
You already paid to see this one

dbt State pays compute only on the days something actually changed. The rest are skipped, because the answer is already sitting in your warehouse.

INSIDE A SINGLE RUN
New data flows downstream
Build what's changed, skip what hasn't.

Zoom into one run. dbt State looks at every model and asks a simple question: has anything that feeds this changed since last time? A new release ripples forward. Everything in its path needs a fresh build. Everything else can be left exactly as it is.

New orders land, so stg_orders and fct_orders rebuild down the chain. mart_revenue would refresh next, but it carries a 48-hour freshness SLA, so dbt State leaves it until the data has to be that fresh. The customer and events branches never changed, so they are reused as-is. One run, almost no extra compute.

HOW IT DECIDES
Three outcomes, decided per model
No manifest scripts, no sub-selectors, no babysitting.

On every run, dbt State compares each model's logic and its upstream data against what it built before. Based on that, it picks the cheapest correct path automatically. You turn it on and it just runs.

  • skipNothing changed. The model's logic and its fresh-enough upstream data match the last build, so dbt leaves the existing table alone.
  • cloneIt exists elsewhere. A matching, fresh copy already lives in another environment, so dbt clones it instead of recomputing it.
  • buildSomething is new. Logic or upstream data changed, so dbt rebuilds this model and everything downstream that depends on it.

It even ignores changes that do not matter, like a new comment or reformatted whitespace, since the compiled logic is identical. A new movie triggers a trip to the theater. A new poster for the same film does not.

WHY IT ADDS UP
The more you run, the more you save
Run as often as the business needs, not as often as the bill allows.

Once you stop paying to redo work that has not changed, frequency stops being the enemy. Teams usually ration runs to control compute. With dbt State, the redundant rebuilds are the part that gets skipped, so running more often mostly adds skips, not cost. You get fresher data and iterate faster, without the bill climbing in lockstep.

Compare→Reuse→Save

That is the whole idea: build what's changed, skip what hasn't. New data flows downstream and triggers a rebuild along its path. Everything else is reused. The more often you run, the more redundant work there is to skip, and the more it pays off.

If you want the relatable version: it is a bit like my Alamo Drafthouse movie pass. The value is not in any single trip, it is in how the savings compound the more you actually use it. Past a certain point of activity, it more than pays for itself, and you stop thinking about cost per visit at all.
