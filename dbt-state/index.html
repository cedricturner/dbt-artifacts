state-aware orchestration, and what it saves

dbt State, explained

TL;DR

By default, dbt rebuilds every model on every run, even the ones whose data did not change. dbt State rebuilds only what changed and skips the rest, so you get the same correct tables for a fraction of the compute. The more often you run, the more you save.

You would not pay to reprint a 400-page report that has not changed one word, just because someone reopened the file. You print again when the pages have actually changed.

By default, dbt does not work that way. Every time it runs, it rebuilds every model, whether or not anything upstream actually changed. At scale, with jobs running many times a day to keep dashboards fresh, that is a lot of money spent reprinting pages that never changed.

dbt State fixes that. Its whole idea fits in six words: build what's changed, skip what hasn't.

INSIDE A SINGLE RUN
New data flows downstream
Build what's changed, skip what hasn't.

Zoom into one run. dbt State looks at every model and asks a simple question: has anything that feeds this changed since last time? New orders ripple forward. Everything in its path needs a fresh build. Everything else can be left exactly as it is.

New orders land, so int_order_items and fct_order_items rebuild down the chain. fct_agg_monthly would refresh next, but it carries a 48-hour freshness SLA, so dbt State leaves it until the data has to be that fresh. The supplier and parts branches never changed, so they are reused as-is. One run, almost no extra compute.

HOW IT DECIDES
Three outcomes, decided per model
No manifest scripts, no sub-selectors, no babysitting.

On every run, dbt State compares each model's logic and its upstream data against what it built before. Based on that, it picks the cheapest correct path automatically. You turn it on and it just runs.

  • skipNothing changed. The model's logic and its fresh-enough upstream data match the last build, so dbt leaves the existing table alone.
  • cloneIt exists elsewhere. A matching, fresh copy already lives in another environment, so dbt clones it instead of recomputing it.
  • buildSomething is new. Logic or upstream data changed, so dbt rebuilds this model and everything downstream that depends on it.

It even ignores changes that do not matter, like a new comment or reformatted whitespace, since the compiled logic is identical. Changed words send the page back to the printer. Reformatting the margins on the same words does not.

WHY IT ADDS UP
The more you run, the more you save
Run as often as the business needs, not as often as the bill allows.

Once you stop paying to redo work that has not changed, frequency stops being the enemy. Teams usually ration runs to control compute. With dbt State, the redundant rebuilds are the part that gets skipped, so running more often mostly adds skips, not cost. You get fresher data and iterate faster, without the bill climbing in lockstep.

Runs in any environment
Dev, staging, or prod, on dbt Core or the dbt platform, across Snowflake, BigQuery, Databricks, and Redshift. Not tied to Fusion, so the whole team gets it.
Freshness SLAs per model
Tell each model how fresh it needs to be. It rebuilds only when it has to, and skips when the data is fresh enough.

SEE IT IN THE PLATFORM
Cost Insights makes the savings a number
From "trust me, it's cheaper" to a figure on the screen.

The skips are the mechanism. Cost Insights is the proof. In the dbt platform it surfaces estimated compute cost at the model and job level, and quantifies the savings dbt State generated by not rebuilding what did not change. That turns platform efficiency into a visible, shareable number.

Compare→Reuse→Save

That is the whole idea: build what's changed, skip what hasn't. New data flows downstream and triggers a rebuild along its path. Everything else is reused. The more often you run, the more redundant work there is to skip, and the more it pays off. Cost Insights then puts a dollar figure on exactly that.

how to turn it on

dbt State runs in dbt Core and the dbt platform with no extra setup. To turn the savings into a number, enable Cost Insights in the dbt platform: it shows cost per model and per job, and the savings from skipping models that did not change.
