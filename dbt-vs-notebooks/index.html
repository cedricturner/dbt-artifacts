import sqlalchemy as sa

engine = sa.create_engine("postgresql://prod-warehouse:5432/db")  ⚠ hardcoded connection, no dev/prod separation

# Step 1: clean customers
tmp_customers = pd.read_sql("SELECT * FROM raw.customers", engine)  ⚠ hardcoded schema — rename breaks silently
tmp_customers['email'] = tmp_customers['email'].str.lower().str.strip()
tmp_customers['first_name'] = tmp_customers['first_name'].fillna('Unknown')
tmp_customers['last_name'] = tmp_customers['last_name'].fillna('Unknown')
tmp_customers['region'] = tmp_customers['region'].fillna('Unassigned').str.upper()

# Step 4: join orders + payments  ⚡ must run cells 2 & 3 first, and hope they were run in that order

summary = tmp_customers.merge(

What you're looking at: One notebook, 5 cells, all data loaded into the kernel, cells that can run in any order, hardcoded connection strings and schema names, full table overwrite on every run, and no trustworthy intermediate state when something fails.     TRIM(LOWER(email))               AS email,
    COALESCE(first_name, 'Unknown')  AS first_name,
    COALESCE(last_name,  'Unknown')  AS last_name,
    CASE WHEN region IS NULL THEN 'Unassigned' ELSE UPPER(region) END AS region,

SELECT
    o.order_id, o.customer_id, o.order_date, o.total_amount,

    COUNT(p.payment_id)                               AS payment_count,

-- Materialization strategy (table, incremental, view) is config, not code.
SELECT
    c.customer_id, c.email,
    c.first_name || ' ' || c.last_name AS full_name, c.region,
    COUNT(op.order_id)                  AS lifetime_orders,
    COALESCE(SUM(op.total_amount), 0)  AS lifetime_revenue,
    COALESCE(SUM(op.total_paid), 0)    AS lifetime_payments,
    MAX(op.order_date)                  AS most_recent_order_date
FROM {{ ref('stg_customers') }} c
LEFT JOIN {{ ref('int_orders_with_payments') }} op
    ON c.customer_id = op.customer_id
GROUP BY c.customer_id, c.email, full_name, c.region
You've now seen all 5 steps. The transformation logic didn't change. What changed: (1) transformations run in the warehouse, not in the notebook kernel, so no memory ceilings and no data transfer overhead; (2) each step is a real, persistent, queryable table — not a variable that vanishes; (3) if_exists='replace' is gone — materialization is a one-line config, not code you have to maintain.
The pattern: Same transformation logic throughout. What dbt changes is the execution environment and the wrapper: transformations move from the notebook kernel into the warehouse, pd.read_sql("... FROM raw.x") becomes source(), tmp_x variable references become ref(). Those three substitutions unlock everything else.
📂
Before the lineage graph makes sense, it helps to understand what the prefixes mean and how dbt tracks where data enters and flows through the pipeline.
stg_
Staging — one file per source table. Cleans, renames, casts. Raw in, usable columns out. Nothing is joined here.
int_
Intermediate — joins and transforms that aren't the final output. Shared building blocks between mart models.
fct_ / dim_
Marts — what BI tools and business users actually query. One table per business question.

source() — where raw data enters

pd.read_sql("... FROM raw.customers", engine)→{{ source('raw', 'customers') }}
pd.read_sql("... FROM raw.orders", engine)→{{ source('raw', 'orders') }}
pd.read_sql("... FROM raw.payments", engine)→{{ source('raw', 'payments') }}
The notebook hardcoded schema names and a connection string. If a schema renames, the cell throws an error, or worse, silently returns stale data. source() registers these as official entry points — dbt can test them, check freshness, and alert you before a bad data day becomes a bad dashboard day. It also resolves the correct schema per environment automatically, so dev and prod never share the same hardcoded string.

ref() — how models depend on each other

tmp_orders (cell variable)→{{ ref('stg_orders') }}
tmp_payments (cell variable)→{{ ref('stg_payments') }}
tmp_customers, tmp_orders_with_payments→ref('stg_customers'), ref('int_orders_with_payments')
Every ref() is a declared dependency. dbt reads them all and builds the execution order automatically. You don't write orchestration code; the ref() calls are the orchestration. In the notebook, the order was enforced by cell position and hope, and running cells out of order quietly broke it; here it's enforced by the system.
This graph is the notebook made visible. The notebook had this same structure, it just wasn't visible. You had to read through five cells (in whatever order they were last run) to understand that step 4 depended on steps 2 and 3. dbt makes it explicit, queryable, and documented automatically.

The notebook runs cell by cell, top to bottom, when you remember to run them in order. That's the only option a kernel gives you. dbt reads the dependency graph and runs independent models simultaneously across threads.

How does dbt know it can parallelize? Because stg_customers, stg_orders, and stg_payments each only reference source() — they have no ref() edges between them. dbt reads the graph, sees no dependencies, and dispatches all three at once. The ref() declarations from tab 2 are the orchestration — you didn't write any extra config to make this happen.
What about scheduling notebooks with papermill or Airflow? You can productionize notebooks with papermill, nbconvert, or an Airflow DAG, but now you're maintaining two systems: the transformation logic in cells, and the schedule and dependency wiring outside the notebook. And a notebook run is still one kernel executing cells in sequence. In dbt, the ref() calls inside your SQL files are the DAG. There's no separate orchestration file to keep in sync.
At scale: This example has 5 models. Real projects have hundreds. A pipeline with 40 independent staging models runs in roughly the time it takes to run one, given enough threads — no orchestrator config required.
🔬
The notebook works until it doesn't. When it breaks, the debugging experience is fundamentally different. Click each scenario to compare.
Pipeline fails at the join step (step 4)▼
notebook

Cell 4 failed. If the kernel is still alive, the DataFrames from cells 1–3 linger, but can you trust them? Were they run in this order, against this version of the code? If you restart the kernel to be sure, they're gone and you re-run every cell from the top. Every debug cycle risks another full re-run, including the expensive pd.read_sql() calls.

dbt

Step 4 failed. stg_orders and stg_payments are still in the warehouse as real tables. Run dbt run -s int_orders_with_payments to re-execute only that model. Upstream work stays intact.

The difference comes from one thing: dbt models are real, persistent tables in your warehouse. Notebook DataFrames live in a kernel and die when it restarts. That's the whole game.
Wrong row count in the final output▼
notebook

Fan-out in the merge? Bad filter in cell 2? You scatter df.shape into extra cells, re-run them, and hope you ran them in the right order against current state. Or you temporarily write intermediate CSVs to disk. Multiple re-run cycles to isolate the problem.

dbt

Query stg_orders — how many rows? Query int_orders_with_payments — how many? The difference shows exactly where the fan-out happened. No re-run needed. The intermediate tables are just sitting there in the warehouse.

Because every intermediate model is a real table, you can bisect a row count problem the same way you'd bisect a bug in code — without re-running anything.
Upstream schema change — a column got renamed▼
notebook

The rename surfaces when you re-run cell 3: KeyError: 'amount'. Cells 1 and 2 already ran and their DataFrames still sit in the kernel, so the notebook is now in a half-updated state. Find every notebook and cell where 'amount' is referenced; there's no central list.

dbt

stg_payments is the only model that touches raw.payments. Fix the column reference there. Every downstream model gets the fix automatically on the next run. One change point, not five notebooks to grep through.

The stg_ layer acts as an isolation boundary. Raw schema changes only affect one file — the staging model for that source.
3am on-call alert fires▼
notebook

The scheduled run failed. It ran in a fresh, headless kernel (papermill), which then exited, so no DataFrames anywhere. Open the notebook, run every cell to reproduce, add cells, run again. Multiple cycles before you even start fixing, and all the expensive source queries run again each time.

dbt

dbt run logs show the exact model that failed and the exact SQL error. Intermediate models are still in the warehouse. Fix the model, run dbt run -s failed_model+ to re-run it and only its dependents.

This is where the value becomes visceral. The engineer on-call at 3am cares a lot about whether the intermediate state survived the failure.
What else depends on payments data?▼
notebook

Grep every notebook for raw.payments, across .ipynb files where the code is buried in JSON cell arrays and hard to search, and hope everyone uses the same connection string format. Hope nobody copied the logic into yet another notebook. No system-of-record for what reads what.

dbt

The lineage graph shows every downstream model that references stg_payments. Click on it in the dbt DAG and see the full impact of any change instantly. A new engineer can answer "what depends on this?" in 5 seconds.

A grep of the codebase finds what you know to look for. The dbt DAG shows you what you didn't know existed.
Reviewing a teammate's change in a pull request▼
notebook

A one-line logic change shows up in the PR as a wall of JSON: shifted execution_count values, regenerated outputs, and base64 image blobs from re-run charts. The actual change is buried in a "source" array. Reviewers can't read it, so they rubber-stamp it or skip review entirely.

dbt

Models are plain .sql files. The same change is a two-line SQL diff in the PR: readable, reviewable, commentable line by line. Version control works the way it does for the rest of your code.

A notebook is JSON with code inside it; a dbt model is code. That's why one reviews cleanly in git and the other doesn't.
The root cause of every dbt debugging advantage: Intermediate models are real, persistent, queryable tables. Notebook DataFrames live in a kernel and disappear when it restarts. Every other difference flows from that one fact.
