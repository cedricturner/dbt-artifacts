From notebooks to dbt

A notebook pipeline, rebuilt in dbt one step at a time.

💡
Below is a five-cell Jupyter notebook that builds a customer order summary in pandas. The highlighted lines point out the operational problems. The In [ ]: counts are out of order because a notebook does not enforce execution order.
build_customer_summary.ipynb: full pipeline notebook 1 notebook · 5 cells

import pandas as pd
import sqlalchemy as sa

engine = sa.create_engine("postgresql://prod-warehouse:5432/db")  ⚠ hardcoded connection, no dev/prod separation

# Step 1: clean customers
tmp_customers = pd.read_sql("SELECT * FROM raw.customers", engine)  ⚠ hardcoded schema — rename breaks silently
tmp_customers['email'] = tmp_customers['email'].str.lower().str.strip()
tmp_customers['first_name'] = tmp_customers['first_name'].fillna('Unknown')
tmp_customers['last_name'] = tmp_customers['last_name'].fillna('Unknown')
tmp_customers['region'] = tmp_customers['region'].fillna('Unassigned').str.upper()

tmp_orders = pd.read_sql("SELECT * FROM raw.orders", engine)  ⚠ hardcoded schema
tmp_orders = tmp_orders[
    tmp_orders['order_id'].notna() & (tmp_orders['status'] != 'cancelled')
][['order_id', 'customer_id', 'order_date', 'status', 'total_amount']]

In [5]:
# Step 3: clean payments
tmp_payments = pd.read_sql("SELECT * FROM raw.payments", engine)  ⚠ hardcoded schema
tmp_payments = tmp_payments[
    tmp_payments['payment_id'].notna() & (tmp_payments['amount'] > 0)
].rename(columns={'amount': 'payment_amount', 'created_at': 'payment_date'})

In [8]:
# Step 4: join orders + payments  ⚡ must run cells 2 & 3 first, and hope they were run in that order
agg = tmp_payments.groupby('order_id').agg(
    total_paid=('payment_amount', 'sum'),
    payment_count=('payment_id', 'count')
).reset_index()
tmp_orders_with_payments = tmp_orders.merge(agg, on='order_id', how='left')
tmp_orders_with_payments['outstanding_balance'] = (
    tmp_orders_with_payments['total_amount']
    - tmp_orders_with_payments['total_paid'].fillna(0)
)

In [11]:
# Step 5: final summary
summary = tmp_customers.merge(
    tmp_orders_with_payments, on='customer_id', how='left'
)
result = summary.groupby(
    ['customer_id', 'email', 'first_name', 'last_name', 'region']
).agg(
    lifetime_orders=('order_id', 'count'),
    lifetime_revenue=('total_amount', 'sum'),
    lifetime_payments=('total_paid', 'sum'),
    most_recent_order_date=('order_date', 'max')
).reset_index()

result.to_sql('fct_customer_order_summary', engine,  ⚠ full overwrite every run, no incremental option
             schema='analytics', if_exists='replace', index=False)

One notebook holds five steps and loads every DataFrame into the kernel. Cells can run out of order, connection details and schemas are hardcoded, each run overwrites the output, and a kernel restart removes the intermediate state. The next tab moves the same work into dbt.
🧩
The notebook keeps five steps in one kernel. dbt puts each step in its own file. Open each block to compare the notebook code with its dbt model.

-- No imports. No engine. Just SELECT.
SELECT
    customer_id,
    TRIM(LOWER(email))               AS email,
    COALESCE(first_name, 'Unknown')  AS first_name,
    COALESCE(last_name,  'Unknown')  AS last_name,
    CASE WHEN region IS NULL THEN 'Unassigned' ELSE UPPER(region) END AS region,
    created_at
FROM {{ source('raw', 'customers') }}
The transformation is now SQL that runs in the warehouse. pd.read_sql("... FROM raw.customers") becomes source('raw','customers'), which tells dbt where the data enters the project. dbt can test that source and monitor its freshness. The model result persists in the warehouse after the run ends.

SELECT
    o.order_id, o.customer_id, o.order_date, o.total_amount,
    SUM(p.payment_amount)                            AS total_paid,
    COUNT(p.payment_id)                               AS payment_count,
    o.total_amount - COALESCE(SUM(p.payment_amount), 0) AS outstanding_balance
FROM {{ ref('stg_orders') }} o
LEFT JOIN {{ ref('stg_payments') }} p ON o.order_id = p.order_id
GROUP BY o.order_id, o.customer_id, o.order_date, o.total_amount

-- Materialization strategy (table, incremental, view) is config, not code.
SELECT
    c.customer_id, c.email,
    c.first_name || ' ' || c.last_name AS full_name, c.region,
    COUNT(op.order_id)                  AS lifetime_orders,
    COALESCE(SUM(op.total_amount), 0)  AS lifetime_revenue,
    COALESCE(SUM(op.total_paid), 0)    AS lifetime_payments,
    MAX(op.order_date)                  AS most_recent_order_date
FROM {{ ref('stg_customers') }} c
LEFT JOIN {{ ref('int_orders_with_payments') }} op
    ON c.customer_id = op.customer_id
GROUP BY c.customer_id, c.email, full_name, c.region
The SQL still performs the same five transformations. dbt runs them in the warehouse, keeps each model available for inspection, and moves the output strategy from if_exists='replace' into model configuration.
The transformation logic stays the same. It moves from the notebook kernel into the warehouse, pd.read_sql("... FROM raw.x") becomes source(), and tmp_x becomes ref(). Those changes give dbt the information it needs to build and run the dependency graph.
📂
The model names describe each layer of the project. source() marks where data enters, and ref() records how models depend on one another.
stg_
Staging: one file per source table for cleaning, renaming, and casting columns. Joins happen later.
int_
Intermediate: joins and transformations used by downstream models.
fct_ / dim_
Marts: tables built for BI tools and business analysis.

source(): where raw data enters

pd.read_sql("... FROM raw.customers", engine)→{{ source('raw', 'customers') }}
pd.read_sql("... FROM raw.orders", engine)→{{ source('raw', 'orders') }}
pd.read_sql("... FROM raw.payments", engine)→{{ source('raw', 'payments') }}
The notebook hardcodes schema names and a connection string. A schema rename can cause an error or return stale data. source() gives dbt a named entry point it can test and check for freshness. It also resolves the correct schema for each environment.

ref(): how models depend on each other

tmp_orders (cell variable)→{{ ref('stg_orders') }}
tmp_payments (cell variable)→{{ ref('stg_payments') }}
tmp_customers, tmp_orders_with_payments→ref('stg_customers'), ref('int_orders_with_payments')
Every ref() declares a dependency. dbt uses those references to build the execution order. In the notebook, execution depends on which cell runs first. In dbt, the graph controls the order.
The notebook contains the same dependencies, but they are implicit in the cell variables. dbt reads each ref() and draws the graph, so the dependency between steps 2, 3, and 4 is visible without reading every file.
⚡
A notebook kernel executes one cell at a time. dbt reads the dependency graph and can run models at the same time when none depends on another.

How does dbt know it can parallelize? dbt can see that the customer, order, and payment models do not depend on one another. That means it can run all three at the same time instead of waiting for each one to finish.
What about scheduling notebooks with papermill or Airflow? Tools like papermill and Airflow can schedule a notebook. You still have to manage the notebook and its run order in separate places. With dbt, the model files describe what depends on what, so dbt can work out the run order for you.
This example has five models. If a project has 40 independent staging models and enough available threads, dbt can start those models together.
🔬
When something breaks, the real difference is how long it takes to understand the problem and move on. Open a scenario to compare the experience.
Pipeline fails at the join step (step 4)▼
notebook

You have to rerun the earlier cells before you can trust what you are looking at. That means waiting for the same work to finish again before you can start fixing the part that failed.

dbt

You open the failed step and inspect the work that led into it. Fix that step, rerun only what needs to run, and move on.

You spend your time fixing the problem instead of rebuilding everything that came before it.
Wrong row count in the final output▼
notebook

You add checks between cells, rerun the notebook, and keep narrowing the problem down. Every guess can mean another wait.

dbt

You check the row count after each step and see where it changed. The results are already there, so you can compare them immediately.

A long cycle of changing, rerunning, and waiting becomes a quick comparison.
Upstream schema change: a column got renamed▼
notebook

You search every notebook for the old column name, change each copy, and rerun them to see what you missed.

dbt

You fix the column name once. Everything that uses that cleaned data picks up the change on its next run.

One fix takes minutes. Finding and updating copies across notebooks can take the afternoon.
3am on-call alert fires▼
notebook

You wake up, open the notebook, and rerun the earlier cells just to recreate the problem. Each attempt can mean another full run.

dbt

You go straight to the failed step, fix it, and rerun that part of the pipeline. Then you go back to sleep.

The goal is a five-minute fix, not an hour spent recreating what happened.
What else depends on payments data?▼
notebook

You search through every notebook you know about and still have to wonder whether someone copied the logic somewhere else.

dbt

You select the payments model and get the full list of work that depends on it.

What used to take a search across the codebase becomes one answer you can trust.
Reviewing a teammate's change in a pull request▼
notebook

A small change can be buried inside a large notebook diff. The reviewer spends time hunting for it or approves it without a close look.

dbt

The reviewer sees the few lines that changed, leaves a comment, and finishes the review.

A review that feels like archaeology becomes a normal code review.
When something breaks, dbt shortens the distance between seeing the problem and fixing it. Less rebuilding. Less searching. Less waiting.
