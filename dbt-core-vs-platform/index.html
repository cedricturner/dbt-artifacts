dbt Core vs. the platform: where the line actually is

A practitioner's guide to the moment dbt Core alone stops being enough, and what you actually get when you start paying for the dbt platform.

where core is still right the tipping points gut-check
the short version

dbt Core is not a limited trial of the paid product. It is the real transformation engine: models, tests, docs, ref(), source(), all of it, and it is free and open source (Apache 2.0) forever. dbt Core is maintained by Fivetran + dbt Labs, following the 2025 merger of the two companies. The combined entity hasn't settled on a permanent name yet, so this page just says "dbt Labs" where it needs to refer to the maker. What the dbt platform adds is not more dbt. It is the operational layer around dbt: the part that runs it on a schedule, decides who can touch what, makes the output findable outside the CLI, and lets more than one project or more than one kind of person work in it at once. For a solo analyst, or a small team that's fine owning all of that itself, dbt Core alone can be genuinely enough. This page maps the specific moments where that stops being true, and what each one actually costs if you keep ignoring it.

⚠ scope note

This draft skips exact pricing, tier names, and plan limits on purpose. The pricing page changes often enough that numbers written here today would likely be wrong by the time anyone reviews this. Fusion, dbt's next-generation engine, gets its own flagged note further down, since its licensing and architecture details are still unsettled as of this draft.

Where dbt Core alone is still the right call

Before any pitch, the honest version. None of the following is dbt Core "settling for less."

  • A solo analyst or a single analytics engineer running their own project against their own warehouse.
  • A small, all-engineer team that's comfortable owning its own scheduler, cron, Airflow, GitHub Actions, whatever it is, and treats that as normal infrastructure work, not overhead.
  • No governance, compliance, or audit requirement forcing a formal, provable record of who changed what and when.
  • One dbt project. Nobody else's project needs to depend on your models, and you don't need anyone else's.
  • Everyone who touches the project is comfortable installing dbt locally, holding warehouse credentials, and working from git and the command line.

If that's your team, nothing below is an argument that you're doing it wrong. It's a map of the specific moments where one of those conditions stops holding, and what quietly starts costing you when it does.

another path worth knowing about

Paying for the dbt platform is not the only way to run dbt somewhere other than your laptop. Snowflake now runs dbt projects natively inside the warehouse, using Snowflake's own compute, another way to get dbt Core's transformation engine working without adopting the dbt platform's operational layer. ⚠ verify before publish, exact feature scope not independently confirmed

Other warehouses may offer similar native or embedded dbt execution paths, worth checking directly. If what you actually need is dbt Core running somewhere more convenient than a laptop, not the platform's scheduling, access control, Catalog, or Mesh, it's worth checking what your own warehouse already gives you before assuming the platform is the only door out.

Where the line starts moving

Each of these starts as a specific, ordinary moment, not a feature you were missing. The cost shows up quietly, in time, risk, or toil, before anyone names what's actually gone missing.

tipping point 1 of 8
Something has to run this on a schedule

It's 2am and nobody is fully sure whether last night's run actually happened.

what core-only costs you

dbt Core has no native scheduler, no native alerting, and no native CI. Someone bolts together cron, or stands up Airflow, or writes a GitHub Actions job to run dbt build on a timer, then owns the credentials, retries, failure handling, and alerting for that glue code themselves, on top of the actual dbt work. Every change to that scheduling layer is now a second codebase that has to stay in sync with the dbt project.

what the platform changes

Hosted job scheduling, run history, and alerting on failure, plus CI checks that run against a pull request before it merges, all run by the platform instead of by whoever's laptop or personal Airflow instance you're currently trusting.

tipping point 2 of 8
More than engineers need to touch it

The one person who actually knows what "a valid order" means keeps sending you SQL over Slack instead of opening a pull request.

what core-only costs you

dbt Core is a CLI. Contributing means a local dbt install, warehouse credentials issued to that person's laptop, and enough comfort with git from a terminal to open a pull request. That's a real barrier for someone whose job isn't engineering, so their domain knowledge stays outside the project, translated secondhand by whoever does have access.

what the platform changes

Studio IDE, a browser-based development environment: no local install, no warehouse credentials to manage on someone's laptop, edit and run models from a browser. It doesn't change who's allowed to approve a merge, but it lowers the barrier to actually contributing in the first place.

tipping point 3 of 8
Who can change what, and who touched this

Someone from compliance asks who can change the model that feeds the board deck, and how you'd prove it if asked.

what core-only costs you

dbt Core has no built-in role-based access control, no SSO, no audit log. Environments (dev, staging, prod) are handled manually through targets in profiles.yml, a convention someone has to maintain and enforce by hand, not something the tool itself enforces. "Who touched this" is whatever git blame happens to say, which tells you who edited a file, not who ran what against production.

what the platform changes

On the platform's higher, Enterprise-tier plans specifically, not every paid tier, RBAC, SSO, and audit logs are available, along with managed dev, staging, and prod deployment environments configured through the platform instead of a hand-maintained profiles.yml.

tipping point 4 of 8
Nobody can find or trust the data

Someone builds a new dashboard on a table nobody remembers owning, and nobody can say with confidence whether it's still good.

what core-only costs you

dbt Core gives you docs and lineage as an artifact you generate and host yourself: a static site, current only as of whenever someone last ran dbt docs generate and pushed it somewhere. There's no ownership signal, no health status, and no always-current view for anyone outside the CLI to search and trust on their own.

what the platform changes

dbt Catalog (formerly dbt Explorer): a governed, always-current lineage and ownership view that doesn't depend on someone remembering to regenerate and redeploy a static site.

tipping point 5 of 8
One repo, many teams

Another team's dbt project needs one of your models, and right now the only way to give it to them is to tell them to copy your SQL.

what core-only costs you

dbt Core has essentially nothing built in for one dbt project to safely depend on another. ref() only resolves inside a single project's own DAG. Teams that need to share models across projects end up copying SQL or duplicating a model into their own project, which quietly creates two definitions of "a valid order" that will eventually disagree with each other, silently, with nothing forcing them to reconcile.

what the platform changes

dbt Mesh. A model gets marked access: public in its config, and other dbt projects can call it directly with a cross-project ref(), the same ref() syntax, now resolved across project boundaries instead of only within one. This requires the upstream project to have a Production deployment environment running on the platform, since cross-project ref() resolves against that project's production artifacts, not whatever happens to be on someone's branch.

see the mesh config →click to expand
upstream project · models/schema.yml
models:
  - name: fct_orders
    config:
      access: public
downstream project · models/finance_summary.sql
select ...
from {{ ref('finance_project', 'fct_orders') }}
access: public isn't a suggestion, it's the switch that lets another project's ref() resolve to this model at all. Anything not marked public stays private to its own project by default.
tipping point 6 of 8
How many people can actually work in this at once

You hire a third analytics engineer, and the onboarding doc for "set up your laptop" is longer than the model they're trying to ship.

what core-only costs you

In a Core-only setup, development is local-only. Every contributor needs their own local dbt install, their own warehouse credentials, their own clone of the repo, and their own way of not stepping on a shared dev schema. There's no hosted place to develop, so how many people can work in the project at once scales with how much environment-setup pain the team is willing to absorb per person, not with how much work there actually is to do.

what the platform changes

The platform's hosted development environment, Studio IDE again, gives every contributor a ready-to-go dev environment without local setup, so headcount on the project isn't bottlenecked by laptop configuration.

tipping point 7 of 8
How many people does the open-source project actually serve

A stakeholder asks to just look at the data model himself instead of filing another ticket, and the honest answer is he can't, not without someone standing up a website for him first.

what core-only costs you

dbt Core's value is largely confined to whoever is running the CLI, the engineer or engineers actually building the project. Non-engineers can only see the output at all if someone manually generates a static docs site and hosts it somewhere, S3, Netlify, wherever, and that site is only as current as the last person who remembered to regenerate and redeploy it. There's no governed, always-current, cross-project way for a broader org to self-serve.

what the platform changes

The platform serves that same documentation and lineage as a governed, hosted, always-current product, dbt Catalog again, so the value of the dbt project reaches past the handful of people who can run a CLI.

tipping point 8 of 8
A metric means three things in three tools

Finance says revenue is 4.2 million, the BI tool says 4.4 million, and both people are confident.

what core-only costs you

You can define metrics locally in dbt Core with MetricFlow, that part is free and works entirely in the CLI. What Core can't do is serve those definitions to anything else. There's no built-in way for a BI tool, a spreadsheet, or an AI agent to query a governed metric and get the same number every time, so every consuming tool ends up re-deriving "revenue" from the tables directly, which is exactly how three tools end up with three answers.

what the platform changes

The platform's Semantic Layer makes MetricFlow-defined metrics queryable by other tools, so "revenue" gets computed once and served consistently everywhere it's asked for.

One more thing, lightly: dbt Labs has said a next generation of dbt Core, referred to as dbt Core v2, is coming. Specifics, timing, and exactly what changes are still emerging, so this page describes the dbt Core you can install today, not a future version of it.

⚠ verify before publish

dbt is also rolling out a next-generation engine called Fusion. How Fusion fits into the Core versus platform line, its licensing, its architecture, and its relationship, if any, to dbt Core v2 above, is still settling. Two of dbt's own docs pages currently disagree with each other on parts of this. Worth checking current docs directly before letting Fusion inform a decision here.

A quick gut-check

Seven honest yes-or-no questions. No question is a trick, and answering "no" to all of them is a perfectly good outcome.

Are you babysitting a scheduler, cron, Airflow, GitHub Actions, whatever it is, just to make sure dbt actually runs?
Do people who aren't engineers need to build or review models, not just look at a finished dashboard?
Would you struggle to prove who ran what against production if someone asked you to, right now?
Do people outside the immediate dbt team need to find or trust a model without filing a ticket to your team first?
Do you have more than one dbt project that needs to reference another project's models?
Are more than two or three people trying to actively develop in the same dbt project at the same time?
Does more than one tool, BI, a spreadsheet, an AI agent, need to query the same metric and land on the same number?

None of this is about whether dbt Core is good. It clearly is, it's the reason any of this works at all. The question this page is actually trying to help answer is narrower: are you currently paying the operational cost of running dbt yourself, scheduling, access control, discoverability, cross-project sharing, concurrency, metric consistency, in time and risk instead of in a subscription. Sometimes that trade is the right one. This is meant to make it a conscious trade instead of an accidental one.
