dbt Core vs. dbt Platform

dbt Core vs. dbt Platform

A practitioner's guide to when dbt Core stops being enough, and what the dbt platform adds.

where core is still right the tipping points gut-check
the short version

dbt Core is the real transformation engine, not a limited trial: models, tests, docs, ref(), source(), free and open source (Apache 2.0) forever, maintained by Fivetran + dbt Labs. The platform doesn't add more dbt. It adds the operational layer around it: scheduling, access control, discoverability, cross-project sharing. For a solo analyst or a small all-engineer team, Core alone can be enough. Below: the specific moments where that stops being true.

Where dbt Core alone is still the right call

Start with the teams that do not need the platform.

  • A solo analyst or a single analytics engineer running their own project against their own warehouse.
  • A small, all-engineer team that's comfortable owning its own scheduler, cron, Airflow, GitHub Actions, whatever it is.
  • No governance or audit requirement forcing a formal, provable record of who changed what and when.
  • One dbt project. Nobody else's project needs to depend on your models, and you don't need anyone else's.
  • Everyone who touches the project is comfortable installing dbt locally and holding warehouse credentials.

If that's your team, nothing below means you're doing it wrong. It's a map of when one of those stops holding.

another path

Paying for the platform isn't the only way to run dbt off your laptop. Snowflake now runs dbt projects natively on its own compute, using its own virtual warehouses with no separate dbt runtime or licensing required.

Other warehouses may offer similar native paths too, so don't assume the platform is the only door out.

dbt Core v2 shipped its first alpha in June 2026. The Rust-based rewrite remains Apache 2.0 licensed. It is not the default stable release yet, so this comparison describes the dbt Core (v1.x) most teams install today.

dbt Core v2 and the proprietary dbt Fusion engine now share that same Rust runtime: Core v2 is the open-source foundation, and Fusion is a separately licensed distribution built on top of it, adding SQL comprehension, column-level lineage, and richer local development. Fusion is free to use locally (via its VS Code extension) and also selectable inside the dbt platform.

A quick gut-check

Seven yes/no questions. Answering "no" to all of them is a perfectly good outcome.

Are you maintaining cron, Airflow, GitHub Actions, or another scheduler to make sure dbt runs?
Do people who aren't engineers need to build or review models, not just look at a finished dashboard?
Would you struggle to prove who ran what against production if someone asked you to, right now?hard requirement
Do people outside the immediate dbt team need to find or trust a model without filing a ticket to your team first?
Do you have more than one dbt project that needs to reference another project's models?hard requirement
Are more than two or three people trying to actively develop in the same dbt project at the same time?
Does more than one tool, BI, a spreadsheet, an AI agent, need to query the same metric and land on the same number?hard requirement
recommendation
Core
Core is the default here. Nothing has pushed you off it yet.

dbt Core makes all of this possible. The choice is whether to pay the operational cost of running it yourself in time and risk, or pay for the platform. Either trade can be right; make it deliberately.
